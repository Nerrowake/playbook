# Releases

Releases should be understandable, repeatable, and honest about impact.

## Release Notes

Release notes should include:

- version or release name
- date
- summary
- added behavior
- changed behavior
- fixed issues
- breaking changes
- upgrade notes
- known issues

## Changelog

Maintain a `CHANGELOG.md` for active projects. Keep entries user-facing and
specific.

Suggested sections:

- Added
- Changed
- Fixed
- Deprecated
- Removed
- Security

## Versioning

Use semantic versioning when the project exposes a package, API, or stable public
contract. For early tools, date-based or milestone releases may be enough.

## Release Checklist

- relevant tests pass
- documentation is updated
- changelog is updated
- version is updated when applicable
- security and privacy impact is reviewed
- migration notes are written when needed
- release notes are clear
