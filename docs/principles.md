# Principles

These principles apply across Nerrowake repositories.

## Clarity Over Cleverness

Code, interfaces, and documentation should be understandable. A clever solution
is only useful when it remains readable and maintainable.

## Maintainability Over Novelty

Choose boring, proven patterns unless the problem clearly requires something
else. New tools and abstractions should reduce real complexity.

## Correctness Before Performance

Make behavior correct first. Optimize when there is evidence, a measurable
constraint, or a clear user impact.

## Documentation as Part of the Product

Documentation is not cleanup work. It is how users, contributors, and future
maintainers understand what the product does and how to operate it.

## Thoughtful Design

Design should support the workflow. Interfaces should be clear, fast, accessible,
and trustworthy.

## Practical Engineering

Prefer solutions that can be built, tested, shipped, and maintained. Avoid
process and architecture that create more work than they remove.

## Long-Term Ownership

Every project should be understandable after time has passed. Important decisions
should be documented, and maintenance should be treated as part of the work.
