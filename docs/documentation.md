# Documentation

Documentation is part of the product. It should help users and maintainers do
real work without guessing.

## Documentation Types

Use the right document for the job:

- README for orientation and quick start
- docs for durable guides and reference
- changelog for release history
- roadmap for planned direction
- comments for local code context
- ADRs when a decision needs a durable record

## README Standard

A project README should include:

- project name
- short description
- status
- installation
- usage
- configuration
- development commands
- testing
- documentation links
- support and security links

## Writing Style

- be direct
- prefer concrete examples
- name prerequisites
- include commands that can be copied
- explain expected output when helpful
- keep docs current with behavior

## Documentation Changes

Update documentation in the same pull request as behavior changes. If docs are
deferred, the pull request should explain why and create follow-up work.
